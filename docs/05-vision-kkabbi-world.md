# 장기 비전 — 까비 월드 (KKABBI WORLD)

> 이 문서는 프로젝트의 **최종 목적지**다. 단기 작업 기준은 여전히 `01-PRD.md`(Phase 0)이며,
> 모든 Phase 0 작업은 이 비전으로 가는 길 위에 있어야 한다.

## 1. 한 문장 정의

**까비(KKABBI) IP의 팬들이 실시간으로 모여, 각자 빌드한 까비 캐릭터로 월드를 돌아다니며
숨겨진 재미를 찾고 기록을 남기고 채팅으로 소통하는 2.5D 소셜 월드.**

학생(제작자)이 원하는 것:
1. 실시간 멀티 — 유저들이 같이 접속해 움직이고 미니게임을 즐기며 **채팅으로 소통**
2. **까비 캐릭터 IP**로 팬들과 소통하는 공간 (게임은 수단, IP 소통이 목적)
3. 게이미피케이션 요소
4. 학생B의 홈페이지에 **연결/임베딩**되어 최종 배포

선생님의 방향: 2005년 클래식 메이플("메이플 플래닛") 감성의 **2.5D 사이드뷰 + 오픈 월드맵** —
캐릭터를 빌드하고, 자유롭게 돌아다니고, 개발자가 숨겨놓은 코드·미션·보물을 찾는다.

## 2. 레퍼런스 분석 — 왜 "2005 메이플 감성"인가

참고: 옛날메이플 트롤플래닛 포스팅 (blog.naver.com/mamlew01/220563920230)

클래식 메이플에서 배워올 핵심은 그래픽이 아니라 **"마을이 곧 소셜 공간"**이라는 구조다:
- 마을(커닝시티 PC방 같은)을 그냥 **돌아다니는 것 자체가 콘텐츠** — 목적 없이 어슬렁거려도 재밌다
- 유저들이 서로 보이고, 말풍선 채팅이 머리 위에 뜨고, 상점·거래·선물이 관계를 만든다
- 숨겨진 맵·아는 사람만 아는 장소가 "여기 가봤어?"라는 대화를 만든다 (입소문 = 팬 소통)
- 2.5D 느낌 = 사이드뷰 + 발판 층(다층 지형) + 전경/배경 패럴랙스 — **현재 까비 코드의 연장선**

→ 결론: 아이소메트릭 전환 불필요. 지금의 메이플식 사이드뷰를 다듬고 깊이감(레이어)을 더한다.

## 3. 핵심 경험 루프

```
캐릭터 빌드(까비 파츠 조합) → 월드 입장 → 자유 로밍(다른 유저가 보임)
   ↑                                            ↓
기록·자랑(방명록/족적/배지) ← 발견(보물·숨겨진 코드·미션) ← 채팅·소통
```

## 4. 단계별 로드맵 — 욕심을 단계로 바꾼다

> 실시간 멀티는 어렵다. 그래서 **혼자서도 재밌는 월드를 먼저** 만들고, 거기에 사람을 넣는다.
> 각 Phase는 그 자체로 완성된 제품이다 (중간에 멈춰도 작품이 남게).

| Phase | 이름 | 내용 | 완료 기준 |
|-------|------|------|----------|
| **0** | 기본기 (= PRD M1~M4) | 저장·게임필·모바일·배포 | 지금 게임이 "완성된 싱글 게임"이 됨 |
| **1** | 까비 월드 (싱글) | 로비 → **월드맵**으로 확장 (마을 2~3개 + 연결 지형), **트레이트 캐릭터 빌더**(아래 §8), **인증샷 부스**, **숨겨진 요소**(보물·코드 입력·미션), 족적/방명록(localStorage) | 혼자 30분 어슬렁거려도 재밌음 |
| **2** | 사람이 보인다 | **실시간 접속자 표시**(다른 까비가 움직임) + **말풍선 채팅** + 방명록 서버화 | 친구 2명과 같은 맵에서 채팅 |
| **3** | 팬 플랫폼 | 학생B 홈페이지에 **임베드**, 닉네임 연동, 미니게임(RETRO CHASER를 월드 안 오락실로!), 배지·칭호 | 홈페이지 방문자가 곧 월드 주민 |

## 5. 기술 전략

### 클라이언트 (Phase 1) — 현재 기술의 연장
- Vanilla Canvas 유지 (03 문서의 결정 그대로) + 레이어 렌더링(배경/지형/유저/전경)으로 2.5D 느낌
- 월드맵 = 현재 LOBBY/던전 데이터 구조의 확장 (`stages.js` → `world/` 맵 데이터)
- 캐릭터 빌더 = 기존 4종 PNG를 **파츠 레이어**(몸/표정/모자/소품)로 분해해 조합 — IP 확장의 핵심

### 실시간 (Phase 2) — 서버 선택

> **구현 기록**: 채팅 MVP는 **Trystero(P2P WebRTC)** 로 먼저 출시했다 — 서버·계정·키가 전혀 필요 없고
> 메시지가 어디에도 저장되지 않아 "접속자끼리만" 요구에 정확히 맞는다. 위치 동기화(다른 까비 보이기)나
> 방명록 등 **상태 저장이 필요해지는 시점에** 아래 비교대로 Supabase를 도입한다.
| 후보 | 장점 | 단점 | 판정 |
|------|------|------|------|
| **Supabase Realtime** | 서버 코드 불필요(presence+broadcast 내장), 무료 티어, **학생B 웹앱과 학습 시너지** | 고빈도 위치 동기화엔 한계 | ✅ **Phase 2 채택** |
| Socket.io + Node 서버 | 자유도 최고, 자료 풍부 | 서버 운영·배포를 학생이 감당해야 함 | Phase 3+ 재평가 |
| Colyseus / PartyKit | 게임 전용 룸·상태 동기화 | 학습 곡선, 한국어 자료 적음 | 유저 늘면 재평가 |

- 동기화 전략: 위치는 초당 5~10회 broadcast + 클라 보간(interpolation). 동시 접속 ~10명 규모면 충분
- **권위(authority)는 욕심내지 않는다**: 치트 방지 서버 검증은 범위 밖 — 팬 커뮤니티 규모에선 신뢰 기반

### 학생B 홈페이지 연동 (Phase 3)
- **iframe 임베드** 채택: 게임은 정적 호스팅 독립 유지, 학생B Next.js에 `/world` 페이지로 iframe
- 닉네임/프로필은 `postMessage`로 전달 (홈페이지 로그인 → 월드 닉네임 자동)
- 이유: 두 학생의 배포 주기가 서로를 막지 않고, 각자 저장소의 독립성 유지

### 운영·안전 (Phase 2부터 필수)
- 채팅: 닉네임 규칙 + 금칙어 필터(클라+서버) + 도배 제한(쿨다운). **사람이 모이는 순간 안전이 기능보다 먼저**
- 비용 0원 유지: GitHub Pages(클라) + Supabase 무료 티어(실시간·방명록)

## 6. 게이미피케이션 설계 (Phase 1부터 심는다)

| 요소 | 구현 | 노림수 |
|------|------|--------|
| 보물찾기 | 맵 구석에 숨겨진 오브젝트, 찾으면 수집됨 | 구석구석 돌아다닐 이유 |
| 숨겨진 코드 | 특정 장소에서 코드 입력(개발자 이스터에그) | 팬 커뮤니티에 "암호" 입소문 |
| 미션 | NPC 까비가 주는 간단한 심부름·퍼즐 | 첫 방문자의 동선 안내 |
| 족적·방명록 | 맵에 내 기록이 남음, 남의 기록이 보임 | 비실시간에도 "사람이 다녀간" 온기 |
| 배지·칭호 | 수집·미션 달성 시 부여, 캐릭터 옆 표시 | 자랑 = 재방문 동기 |

## 7. IP 전략 (게임보다 오래 남는 것)

- 까비 캐릭터 4종(Machaei·Mahanni·Mareh·Marisa)의 **세계관 한 페이지 문서**를 만든다 — 이름·성격·관계.
  파츠 빌더도, 미션 대사도, 팬 소통도 전부 이 문서에서 나온다
- 월드는 IP의 "본진", 학생B 홈페이지는 "공식 홈" — 팬아트·공지 등 커뮤니티 기능은 홈페이지 영역

## 8. NFT 연동 — SpaceKkabbi (opensea.io/SpaceKkabbi)

까비 IP는 OpenSea의 **SpaceKkabbi NFT 컬렉션**과 연결돼 있다. NFT 프로젝트의 핵심 구조인
**트레이트(trait) 시스템**을 캐릭터 빌더의 설계 기준으로 삼는다 — 게임과 NFT가 같은 언어를 쓰게.

### 트레이트 캐릭터 빌더 (Phase 1)
- 파츠 카테고리: **헤어 / 얼굴(표정) / 의상 / 아이템 / 신발 / 배경(오라)** — NFT 컬렉션의 트레이트 구분을 따른다
- 데이터 구조도 NFT 메타데이터와 호환되게: `[{ trait_type: "Hair", value: "Neon Green" }, …]`
  → 나중에 실제 NFT와 1:1 대응 가능, 인증샷에 트레이트 목록 표기 가능
- 기존 캐릭터 PNG 4종은 "프리셋"으로 유지하고, 빌더는 파츠 레이어를 겹쳐 그리는 방식

### NFT 연동 수준 (단계적·안전 우선)
| 레벨 | 내용 | 시기 |
|------|------|------|
| L0 | 월드/홈페이지에 SpaceKkabbi 컬렉션 링크·갤러리 표시 | Phase 1 |
| L1 | (선택) 지갑 **주소만 입력** → OpenSea API로 보유 NFT 조회 → 해당 트레이트 파츠 잠금해제 | Phase 3 |
| ❌ | 지갑 서명·트랜잭션·민팅을 게임 안에서 직접 다루는 것 | 하지 않는다 (보안·복잡도·연령 고려) |

### 인증샷 부스 (Phase 1 — 작고 임팩트 큰 기능)
- 월드 안 포토존(또는 P키)에서 **내 캐릭터 인증샷을 PNG로 다운로드**
- 구현: `canvas.toDataURL()` + 프레임·닉네임·날짜·트레이트 목록 합성 — 약 30줄
- 노림수: 인증샷이 SNS·커뮤니티에 퍼지는 것 자체가 IP 홍보 (팬 소통 목적과 직결)

## 9. 하지 않을 것 (전체 기간)

- 계정/비밀번호 시스템 자체 구현 (닉네임만, 필요 시 학생B 홈페이지 인증에 위임)
- 아이템 거래·경제 시스템 (클래식 메이플의 추억이지만 운영 부담이 감당 불가)
- 네이티브 앱 (웹으로 충분)
- Phase 0~1을 건너뛴 멀티플레이어 선행 개발 — **혼자서 재미없는 월드는 같이 있어도 재미없다**
